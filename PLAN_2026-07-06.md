# Thousand-Brains Architecture Update — Design Decisions

**Date:** 2026-07-06
**Status:** Decisions validated — awaiting implementation of Step 1
**Goal:** Move ACN closer to Thousand Brains theory — varying receptive fields
(4×4 / 6×6 / 6×8), binary on/off gate, only the most active columns fire.

---

## Context

Current ACN: 64 columns, all see the same 7×7 patch on a regular grid, soft
gate in [0,1]. We want to evolve it toward Thousand Brains:

- Mini columns (mini networks) have **varying tiny receptive fields** on the
  image (4×4, 6×6, 8×8).
- Each column has a **0/1 weight** (on or off, not soft).
- Only the **most active** columns fire (top-k).

This document records the 5 design decisions made this session, the options
considered, and the chosen path. Implementation will be done in two steps:

- **Step 1:** Multi-scale decomposition (Decisions 1, 2, 3) — patches of
  different sizes, fed through one shared encoder.
- **Step 2:** Binary top-k gate (Decisions 4, 5) — separate, so we test one
  change at a time.

---

## Decision 1: How do columns get their "view" of the image?

**Question:** When a new image comes in, do the columns' positions/sizes stay
fixed or move around?

### Options considered

- **A — Fixed assignment:** every column has a permanent identity
  ("column #5 = 4×4 patch at top-left"), set once at init, same for every image.
  Brain-like (cortical columns don't move). Simple, deterministic, debuggable.
- **B — Random per forward pass:** sample positions/sizes fresh each image.
  A form of augmentation, but complicates evaluation.
- **C — Learned:** the model figures out where to look. A research project on
  its own — too much for this step.

### Decision: **A — Fixed assignment**

Brain-like, simplest, deterministic. Columns have permanent identity.

---

## Decision 2: What patch sizes, and how many columns?

**Question:** Which receptive-field sizes do the columns see, and how many
columns total?

### Options for sizes

- {2×2, 4×4, 6×6} — very fine, but 2×2 is probably too tiny to be useful on a
  28×28 digit (like looking at 2 pixels).
- **{4×4, 6×6, 8×8}** — small to large. 4×4 sees fine stroke detail, 8×8 sees
  a big chunk (whole top of a digit). Good range.
- {2×2, 4×4, 6×6, 8×8} — everything, but 2×2 likely noise.

### Options for count

- **Keep 64 columns** — apples-to-apples with the current model.
- More (96, 128) — richer coverage, slower, more params.
- Fewer (32) — faster but poor coverage with varied sizes.

### Decision: **~64 columns, mix of {4×4, 6×6, 8×8}, roughly equal counts**

- 64 keeps comparison with the current model clean.
- {4, 6, 8} covers fine detail AND big picture without the probably-useless 2×2.
- Equal mix is the simplest starting point (~21 of each size = 63 total).
- Made a config option so we can tune the distribution later.

---

## Decision 3: How do we feed different-sized patches into one encoder?

**Question:** The encoder expects a fixed input size. How do we handle patches
of different sizes?

### Options considered

- **A — Adaptive-pool every patch to a common internal size** (e.g. 4×4), then
  one shared encoder. Resize all photos to the same album size. Keeps one
  encoder, shares learning across sizes. Standard PyTorch one-liner
  (`F.adaptive_avg_pool2d`).
- **B — A different encoder per patch size** — three encoders. More
  biologically faithful but ~3× the parameters and complexity.
- **C — Pad small patches up to the biggest size** — wasteful, encoder
  processes lots of zeros.

### Decision: **A — Adaptive-pool to fixed internal size, one shared encoder**

- One shared encoder — simpler, fewer params, shares learning across sizes.
- The "what size did this column see" info is preserved in the column's identity
  (column #5 is "the 8×8 one at position X"), not lost.
- The slight detail loss from shrinking big patches is fine — big patches are
  about the big picture, not pixel-perfect detail.
- Option B kept as a fallback if A underperforms.

---

## Decision 4: How does the gate become binary (on/off)?

**Question:** The gate is currently soft [0,1]. We want hard 0/1. But gradient
descent needs smooth functions — "pick 0 or 1" has zero gradient.

### Options considered

- **A — Soft during training, hard during inference.** Train with the soft gate
  (gradients flow), snap to hard top-k when using the model. No new math.
- **B — Hard during training too** (Gumbel-softmax / straight-through estimator).
  Truly hard but the tricks are finicky and can bias gradients.
- **C — Keep soft, squeeze toward binary with a stronger penalty.** Lazy option,
  doesn't really commit to the binary idea.

### Decision: **A — Soft during training, hard top-k during inference**

- Simplest — no new math, training loop unchanged.
- Model already trains fine with soft gate; just add hard top-k at inference.
- We can compare soft vs hard outputs to see if hard actually helps.
- Cleanly separates "how we train" from "how we decide."
- Option B reserved for later if A doesn't give the sparsity we want.

**Note:** This is Step 2 — separate from Step 1 so we test one change at a time.

---

## Decision 5: "Most active" means top-k, or a threshold?

**Question:** Once the gate gives relevance scores, how do we decide which
columns fire?

### Options considered

- **A — Top-k (always fire exactly k columns).** Sort by relevance, pick top k
  (e.g. k=8), fire those. Deterministic count.
- **B — Threshold (fire everyone above a bar).** Variable count per image. More
  brain-like but harder to train — model has to learn both *which* and *how many*.

### Decision: **A — Top-k (k=8)**

- Deterministic — always know how many columns fire. Easy to debug.
- Guarantees sparsity — k=8 means exactly 8 columns, period.
- More stable training — model only learns "which k are best," not "how many."
- Matches "only the most active fire" exactly.
- Option B (threshold) is more brain-like but wrong starting point; can revisit
  later if top-k feels too rigid.

**Note:** This is Step 2 — paired with Decision 4.

---

## Summary — chosen package

| Decision | Chosen | One-line reason |
|---|---|---|
| 1. Column placement | **Fixed assignment** | Brain-like, simple, deterministic |
| 2. Sizes & count | **64 columns, {4×4, 6×6, 8×8}, equal mix** | Good range, apples-to-apples with current |
| 3. Encoder input | **Adaptive-pool to fixed size, one shared encoder** | One encoder, shares learning across sizes |
| 4. Binary gate | **Soft train, hard top-k at inference** | Simplest, no math tricks, trains fine |
| 5. Who fires | **Top-k (k=8)** | Deterministic, guarantees sparsity |

## Implementation plan (high level)

### Step 1 — Multi-scale decomposition (Decisions 1, 2, 3)
Implement and test first. Changes:
- New `decompose_multi_scale()` in `acn/decomposition.py` — extracts patches of
  multiple sizes, returns patches + per-column `(size, location)` metadata.
- Adaptive-pool every patch to a fixed internal size (e.g. 4×4) so the existing
  encoder works unchanged.
- Generalize `build_spatial_neighbors` in `acn/topology.py` to compute overlap
  between different-sized patches.
- Config option: `patch_sizes: tuple = (4, 6, 8)` with a count distribution.
- Gate, encoder, consensus, fusion all unchanged for Step 1.

### Step 2 — Binary top-k gate (Decisions 4, 5)
Implement and test after Step 1 is validated. Changes:
- Keep soft `sigmoid(s)` gate during training (gradients flow).
- Add hard top-k selection at inference: pick top `k=8` columns by relevance,
  set them to 1, others to 0.
- Config options: `gate_hard_infer: bool`, `gate_topk: int = 8`.

## Why we're keeping the encoder + ADMM (not skipping to direct voting)

During discussion, the question came up: "why not skip the encoder and go
straight from patch → vote?" The answer:

- **The encoder IS the mini network.** A column without an encoder is just raw
  pixels — pixels can't form an opinion, decide if they're relevant, or vote.
  The encoder produces both the gate signal (`s_i`) and the ADMM opinion
  (`A_i, b_i`). It's the column's brain.
- **We're keeping ACN (Architecture A), not dropping to a direct-vote ensemble
  (Architecture B).** ADMM negotiation is the thesis of the project — it's what
  makes ACN different from "64 patch classifiers averaged together." Dropping it
  would throw away the Physarum wires, the consensus state, the dual variables,
  the entropy confidence — the entire ACN idea.

So the encoder stays. The "mini" in "mini network" can be made smaller later if
it feels too heavy, but the encoder + ADMM + vote pipeline is the architecture.

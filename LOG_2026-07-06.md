# LOG 2026-07-06 — Entropy fusion + all-pairs topology

Working session on the ACN model's robustness story. Two changes landed:
(1) replacing the broken `‖u_i‖`-based confidence with entropy-based confidence,
(2) adding an `allpairs` topology mode that wires every pair of active columns
regardless of image position. A retrain with the new entropy fuse + all-pairs
topology is in progress.

---

## What we were working on

The advertised "adaptive robustness" mechanism — `confident_fuse` — was supposed
to suppress confused columns on corrupted inputs so the model stays accurate
under distribution shift (rotation, occlusion, noise). It wasn't helping: on the
old checkpoint, `sparse_fuse` and `confident_fuse` produced near-identical
accuracy across all corruption conditions. We diagnosed why, fixed it, and then
attacked a deeper architectural limitation the diagnosis surfaced.

---

## Changes made

### 1. Entropy-based `confident_fuse` (replaces `‖u_i‖`-based)

**`acn/networks.py` — `confident_fuse`**

Old:
```python
u_norm = u.norm(dim=-1)
confidence = torch.exp(-u_norm / tau)      # exp(-‖u_i‖/tau)
```

New:
```python
p = torch.softmax(logits, dim=-1)
H = -(p * (p + 1e-9).log()).sum(dim=-1)    # entropy of the column's own scorecard
H_norm = H / log(C)                         # normalized to [0,1]
confidence = torch.exp(-H_norm / tau)       # sharp → ~1, flat → ~0
```

Signature dropped the `u` argument (no longer needed). The column's confidence
now comes from **how sharp its own per-class prediction is**, not from how much
it fought its neighbors during consensus.

**`acn/model.py`** — call site updated to `confident_fuse(logits, active, tau=...)`.

**`acn/config.py`** — `fuse_mode` / `fuse_tau` docstrings rewritten to describe
the entropy mechanism and why the `u`-based version failed.

Default unchanged: `fuse_mode="confident"`, `fuse_tau=1.0`.

### 2. `allpairs` topology mode (opt-in, default stays `spatial`)

**`acn/topology.py`** — added `build_all_pairs(num_nodes)` returning the complete
graph (every pair `i<j`). For N=64 → E=2016.

**`acn/config.py`** — new field:
```python
topology_mode: str = "spatial"   # "spatial" | "allpairs"
```

**`acn/model.py`** — `_build_topology_from_geometry` branches on `topology_mode`:
- `"spatial"` → original overlap-based edges (546) + overlap `D_init`.
- `"allpairs"` → complete graph (2016) + dense `D_init` (no spatial overlap to
  initialize from; Physarum dynamics decide which wires matter).

The entire consensus/ADMM/Physarum/fusion pipeline downstream is **unchanged** —
it already operated on whatever edge list it was given and already masked by
activity via `edge_mask = active[:, ei] * active[:, ej]`. No algorithm changes.

Usage: `python train.py model.topology_mode=allpairs`

---

## Decisions

1. **Entropy over `‖u_i‖` for confidence.** `u` measures disagreement-with-
   neighbors accumulated over rounds; entropy measures prediction sharpness
   directly. Entropy works for isolated columns (no neighbor needed) and
   discriminates clean vs corrupted patches. See "Lessons learned" for why `u`
   was rejected.

2. **All-pairs wiring as a fixed complete graph + activity masking**, not as
   per-sample dynamic edges. Per-sample edge sets would break the batched
   `index_add_` consensus op (128 different edge lists can't go through one GPU
   kernel). The fixed complete graph (2016 edges) keeps batching and lets
   `edge_mask` filter to the ~6 live wires per input. Cost: ~47k extra params
   (restriction maps) and ~10% slowdown. Accepted as the price of global
   consensus.

3. **Physarum grow/prune runs on ALL edges in all-pairs mode** (no special
   treatment for non-spatial pairs). This is the optimistic version. The risk
   is over-coupling: non-spatial pairs whose restriction maps don't learn
   meaningful correspondences may perpetually "disagree" and saturate to
   `D_clip`, killing ensemble diversity. We did NOT pre-build the safer hybrid
   (Physarum on spatial only, fixed weak wires on non-spatial) because that
   solves a problem we haven't confirmed exists. Decision: run plain all-pairs
   first, fall back only if diagnostics show over-coupling.

4. **`lambda_sparse` not re-tuned.** It multiplies `D.mean()`, a per-edge
   average, so it's automatically scale-invariant to edge count. No change
   needed going from 546 → 2016 edges.

5. **Old broken checkpoint deleted** before retraining:
   `results/runs/model_result/{model_best.pt, robustness.gif, config.yaml,
   D_epoch*.npz}` removed so the retrain starts clean.

## Lessons learned

### The `‖u_i‖` confidence was a dead mechanism — and I misdiagnosed it twice

**First (wrong) diagnosis:** I claimed `u` was ~0 for ~90% of active columns
because the sparse spatial graph left active columns *isolated* (no active
neighbor → no pull on `z` → `x−z≈0` → `u` stays 0). I backed this with a probe
showing `‖u‖` p50 = 0.000.

**The bug in my probe:** I was computing `u_norm * active` (u multiplied by the
continuous gate value in [0,1]) instead of `u_norm` on genuinely active columns.
Because the gate is continuous and most columns have a small gate value, `u ×
gate` looked tiny. I misread that as "u is zero."

**Corrected measurement** (masking by `active > 0.5`, not multiplying):
```
clean:   active=7.0  isolated=0.19  act_edges/col=1.39  ||u||_active p50=17.6 mean=18.6
noise30: active=13.0 isolated=0.44  act_edges/col=1.72  ||u||_active p50=19.6 mean=19.6
```

The real facts: columns are NOT isolated (only ~0.2 of 7 active columns have no
active neighbor), `u` is NOT zero (it's large, ~18–28), and the consensus IS
running. My "isolated columns" story was wrong.

**The real reason `u` doesn't help robustness:** `u` is large on clean AND
corrupted inputs (18.6 vs 19.6 — barely moves). It carries almost no
information about input quality. Why: `u_i += x_i − z_i` every round for 20
rounds, and the consensus never perfectly converges (`‖x−z‖ ≈ 0.16` persists),
so `u` keeps accumulating and ends up dominated by the accumulation dynamics,
not by input quality. `u` is a **low-quality confidence signal**, not a broken
architecture.

**Lesson:** always print the raw masked statistic, not a product with a
continuous mask. And don't build a story on a single suspicious number without
re-deriving it a second way. I should have caught the `u × gate` bug before
narrating "isolated columns."

### ADMM converges, but consensus is local — fusion carries the global decision

Per-round trajectory on the entropy-fuse checkpoint (clean, 95% accuracy):
```
round   ||x-z||   ||z_i-z_j||   vote_agree
   0    7.2662    0.6825        42.6%
  19    0.1162    4.0133        57.9%
final fused entropy 0.114, accuracy 95.0%
```

- `‖x−z‖` shrinks 60× → ADMM converges in the textbook sense. Not a false
  positive.
- `‖z_i − z_j‖` *grows* monotonically 0.68 → 4.01 → neighboring consensus
  states diverge, not converge. The consensus coupling is weak relative to
  each column's self-pull `ρ(x_i + u_i − z_i)`.
- Vote agreement only 58% → by design. Columns see different patches and
  *should* disagree; this is the "diverse ensemble that softly agrees."
- Fused prediction entropy 0.114 (very sharp) + 95% accuracy → fusion turns
  the diverse votes into a crisp correct answer. **Fusion is doing the global
  work; ADMM is only doing local reconciliation.** This is the motivation for
  all-pairs: make ADMM's negotiation global so it shares the load with fusion.

### 58% agreement is not a failure of convergence

It's the ensemble's intended diversity. The 95% accuracy is not a false
positive — the columns converge in the ADMM sense (`x→z`), and fusion resolves
the remaining diversity into one sharp answer. Don't read low vote-agreement as
"columns didn't converge."

## Failed / didn't work

- **`‖u_i‖`-based `confident_fuse`.** Did not improve robustness over
  `sparse_fuse` (tied across all corruption conditions). Root cause: `u` is
  uninformative about input quality — large on clean and corrupted alike
  because it accumulates residual consensus gap over 20 rounds. Replaced with
  entropy-based confidence.

- **My first diagnosis of why `u` failed ("isolated active columns").** Wrong.
  Caused by a bug in my probe (`u_norm * active` instead of `u_norm` on masked
  active columns). The corrected probe showed columns are well-connected and
  `u` is large. The real cause was signal quality, not graph isolation.

- **Old checkpoint (~49% clean accuracy) as a substrate for tuning robustness.**
  The pre-entropy-fuse checkpoint was at ~49% clean MNIST — far too low to make
  any robustness claim meaningful. Deleted; retraining from scratch with the
  entropy fuse. (The entropy-fuse retrain hit 93.9% by epoch 10 and 95% final —
  the fuse fix plus fresh training recovered real accuracy.)

### Immediate (this run, in progress)
- **Let the all-pairs + entropy-fuse retrain finish (50 epochs).** Watch:
  - epoch 10 eval acc vs spatial's 0.9389 at the same epoch (parity check).
  - gate active fraction (should prune to ~6%; if it stays high, z-loss is
    losing → bump `column_z_loss_weight`).
  - zloss (started higher, 6.5 vs 3.8 — keep an eye it doesn't keep climbing).
- **Over-coupling failure check** on the all-pairs run. Diagnose after training:
  - mean `D` on spatial vs non-spatial pairs (if non-spatial saturates at
    `D_clip`, the Physarum rule is over-firing).
  - vote-agreement fraction (if → ~100%, ensemble diversity collapsed).
  - clean accuracy vs spatial baseline (if well below, global coupling hurts).
- **Run `python scripts/eval_robustness.py`** on both spatial and all-pairs
  checkpoints. The robustness delta is the actual verdict on whether all-pairs
  earns its keep.

### Conditional (only if all-pairs helps robustness)
- **Cut the parameter waste.** ~47k of the 115k params are restriction-map
  vectors for far-apart pairs that fire on ~0.36% of inputs and barely train.
  Replace per-edge `RestrictionMaps` with a shared/low-rank parametrization
  (e.g. one `r` per column + a mixing function → pairwise map constructed, not
  stored). Drops ~64k → ~2k params and every param gets gradient on every input
  a column is active (~6%, not 0.36%).
- **If over-coupling appears:** the safer hybrid — Physarum grow/prune on
  spatial edges only, fixed weak conductance on non-spatial pairs. Don't build
  unless the all-pairs run fails the diagnostics above.

### Separate (orthogonal, flagged but not addressed)
- **Tune consensus coupling if we want stronger global agreement.** `‖z_i−z_j‖`
  growing over rounds suggests the consensus pull is weak vs self-pull. Nudge
  `rho` up, `eta_z` up, or `diffusion_steps` up. This is independent of the
  all-pairs change and could be tried on the spatial baseline too.
- **Investigate the low clean-accuracy regime** if it recurs. The old checkpoint
  was at 49% clean; the entropy-fuse retrain reached 95%. Worth understanding
  what made the old one stall — possibly gate collapse, underfitting, or the
  dead confidence mechanism starving the decoder of useful gradient. The
  entropy fix may have fixed this indirectly; confirm on the next run.
